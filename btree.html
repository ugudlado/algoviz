B-Tree

Data Structures O(log n)
Node keys (sorted) Min degree t: max 2t−1 keys per node
Choose minimum degree t, then insert integer keys (−999…999). Up to 30 keys. Step through splits, merges, and borrows after each operation.
Speed

Context

The Problem

Disk blocks and memory pages hold many keys at once. Binary trees are too shallow per block — we want high fan-out and shallow height.

The Approach

Each node stores up to 2t−1 sorted keys and (if internal) 2t child pointers. Overflow splits a node and promotes the median key to the parent; underflow borrows from a sibling or merges.

When To Use

Database indexes, file systems (e.g. ext4 concepts), any on-disk ordered map where sequential reads inside a block are cheap.

Operation colors (steps)

Split — median promoted; merge / borrow — fix underfull child; search — orange highlight on nodes containing the target key.

Pseudocode
insert(root, k, t): if root full: split root, new root insertNonFull(root, k) if leaf: insert k in order else: if child full, split child recurse into correct child delete(root, k, t): find k; borrow/merge so child ≥ t−1 recurse until removed
Real-world Analogy

Real-world analogy: A filing cabinet where each drawer holds several folders (keys) in order. When a drawer is full, you split it into two drawers and move the middle label up to the index tab on the next level — keeping lookup fast without any single drawer holding hundreds of thin folders.
