Context

The Problem

Comparison-based sorts can't beat O(n log n). But what if your data is integers with bounded digits? Can you exploit that structure?

The Approach

Sort by least significant digit first, then next digit, and so on. Each pass is stable and O(n). After d passes (d = max digits), the array is sorted.

When To Use

Integer keys with bounded range (phone numbers, zip codes, timestamps). Beats O(n log n) comparison sorts when d is small relative to n.

Pseudocode
radixSort(arr): d = max digits in arr for pos = 0 to d - 1: buckets = [[] for 0..9] for each num in arr: digit = (num / 10^pos) % 10 buckets[digit].add(num) arr = concat(buckets)
Real-world Analogy

Sorting mail at a post office — first sort all letters by the last digit of the zip code into 10 bins. Collect and re-sort by the next digit, and so on. After processing every digit, all mail is perfectly sorted — without ever comparing two zip codes directly.
